# creator 迁移审计与结构决策

## 结论

上游 `yyy_monkey/creator` **不适合原样合并**，但其“分阶段创作工作台”能力适合迁移到 `yyy_ball`。

更具体地说：

- 适合迁移的是：抓取 -> 分析 -> 人工确认 -> 规划 这套工作流能力。
- 不适合原样合并的是：独立子项目边界、写死的育儿账号语境、与上游认证目录约定耦合的实现。

## 审计结果

### 1. 上游 `creator` 的优点

- 结构简单，只有 `scraper.py` / `analyzer.py` / `planner.py` / `__main__.py` 四个核心模块。
- 流程边界清晰，存在人工确认节点，不会直接让 LLM 输出成为最终结论。
- 抓取和规划之间通过 JSON 报告解耦，可审阅、可复用。

### 2. 原样合并的主要问题

#### 2.1 双项目结构不合理

当前 `yyy_ball` 已经有根级 `pyproject.toml`。如果再把上游 `creator/pyproject.toml` 整体放进来，会形成两个并行 Python 项目，带来：

- 依赖重复维护。
- 启动命令风格不一致。
- 后续升级时更容易漂移。

#### 2.2 内容语义与当前仓库不匹配

上游配置、分析关键词、提示词都围绕“懿起成长”公众号展开，主题偏家庭育儿。  
当前 `yyy_ball` 的主内容是兵法、财商、英语等教育短视频，因此分析器必须改为配置驱动，不能保留上游写死的主题体系。

#### 2.3 认证与路径约定不适合照搬

上游默认把认证状态放在 `~/.media-publisher/`，这是另一个仓库的命名空间。  
迁移后改为 `~/.yyy_ball/creator/`，避免把当前仓库继续耦合到外部系统名。

#### 2.4 当前宿主本身也存在结构漂移

`yyy_ball` 当前 README 仍描述了并不存在的 `src/publish/` 层，说明仓库已有文档/结构漂移。  
如果此时再直接塞入一个原样子项目，只会进一步加剧边界混乱。

## 最终结构决策

### 代码层

- `src/creator/`
  - 放可执行逻辑。
  - 走当前仓库已有依赖环境。
  - 使用 `python -m src.creator` 运行。

### 配置与产物层

- `creator/config/target.json`
  - 放账号定位、栏目体系、分析关键词。
- `creator/output/`
  - 放抓取结果、分析报告、规划结果。

### 为什么这样更合理

- 不污染 `series/`。`series/` 继续只承载 lesson 内容与渲染产物。
- 不把 creator 绑进 Manim runtime。策划层和渲染层继续分离。
- 保持单仓单环境，后续更容易演进成更通用的创作前置工作台。

## 本次迁移时做的反思与修正

### 修正 1：去掉对 `rich` 的额外依赖

上游 CLI 使用 `rich` 美化输出，但当前仓库并不需要为了一个小型工具增加额外依赖。  
迁移后改成标准库日志与普通终端输出。

### 修正 2：把分析器改成配置驱动

上游分析器把主题关键词与实体模式写死。  
迁移后允许通过 `creator/config/target.json` 覆盖主题分类与实体识别规则。

### 修正 3：给规划器增加离线 mock 模式

上游 `planner.py` 只能真实调用 LLM，不利于自动测试。  
迁移后新增 `--mock-response-file`，可以在没有 API key 的情况下跑通完整规划流程。

### 修正 4：给规划结果增加结构校验

新增数量校验、重复标题校验、字段完整性校验，避免把一份格式不完整的 LLM 输出当成成功结果。

## 当前仍保留的限制

- 抓取器仍然特定于微信公众号后台。
- 规划器仍然输出“日历式内容规划”，还没有直接生成 `script.json`。
- creator 目前是策划前置工作台，而不是 lesson 脚手架生成器。

## 下一步如果继续演进

最值得继续做的不是扩充抓取器，而是把 creator 的规划结果继续映射到当前仓库现有生产链路：

1. 规划结果 -> `script.json` 草稿。
2. `script.json` 草稿 -> `animate.py` 模板。
3. creator 输出与 `series/prompts/*.prompt` 联动。

这样 creator 才会真正成为 `yyy_ball` 的创作前台，而不是一个独立附属工具。
